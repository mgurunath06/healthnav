## 1. System Overview

### 1.1 What HealthNav does
User describes a symptom in plain English. System runs a **dynamic chain of AI agents** — the path is decided at runtime by a Supervisor based on intermediate findings — and produces a **Doctor Prep Card**: a structured summary the user can take to their doctor visit.

### 1.2 Product positioning (legal/ethical)
HealthNav is a **preparation tool**, not a diagnostic tool. It exists to help users organize their thoughts before a doctor visit. It does not diagnose, prescribe, or replace medical advice. Every output recommends consulting a licensed medical professional.

### 1.3 Data flow
```
[User]
  → POST /investigate { symptom_description }
  → Backend orchestrates:
      ├─ Guardrail Agent (parallel, always — anti-gaming, anti-misuse)
      ├─ Triage Agent (parallel, always)
      ├─ Red Flag Detector (parallel, always)
      ├─ Symptom Deep-Dive (after Triage, conditional on Guardrail pass)
      ├─ Lifestyle Agent (conditional, decided by Supervisor)
      └─ Assembler (always last)
  → Returns Doctor Prep Card JSON OR Redirect OR Emergency Advisory
[Frontend] renders appropriate screen
```

### 1.4 Key architectural decisions
- **Stateless backend.** Each `/investigate` call carries everything it needs. No DB, no session.
- **Anonymous use.** No accounts, no PII storage in v1.
- **Single endpoint.** `/investigate` runs the full flow.
- **Streaming optional, not v1.** Frontend waits for response.
- **Disclaimers mandatory.** Every Doctor Prep Card includes a non-diagnostic disclaimer.
- **Correlation-ready architecture.** Structured logs with `request_id` from day one. v3 (correlation features) plugs into this without rewriting.

## 2. API Contracts

### 2.1 `POST /investigate`

**Request:**
```json
{
  "request_id": "string, UUID v4, generated by frontend",
  "symptom_description": "string, 10-2000 chars, required",
  "follow_up_answers": {
    "question_id": "answer_value"
  }
}
```

`follow_up_answers` is optional. On the first call, it's empty `{}`. If the system needs more info, the frontend re-calls `/investigate` with the same `request_id`, same `symptom_description`, plus filled-in answers.

**Response (success — final card ready):**
```json
{
  "status": "complete",
  "request_id": "string, echoed",
  "doctor_prep_card": { /* see Section 5 */ },
  "agent_trace": [ /* see Section 6 */ ]
}
```

**Response (needs follow-up):**
```json
{
  "status": "needs_followup",
  "request_id": "string, echoed",
  "questions": [ /* see Section 3.4 — options-based */ ],
  "agent_trace": [ /* partial trace */ ]
}
```

**Response (emergency):**
```json
{
  "status": "emergency",
  "request_id": "string, echoed",
  "advisory": "Seek immediate medical attention. [reason]",
  "red_flags": ["chest_pain", "..."],
  "agent_trace": [ /* partial trace */ ]
}
```

**Response (redirect — Guardrail triggered):**
```json
{
  "status": "redirect",
  "request_id": "string, echoed",
  "message": "We're not able to help with this query. Please consult a healthcare professional for personalized advice.",
  "reason_category": "out_of_scope | unclear_input | prompt_injection | seeking_diagnosis",
  "agent_trace": [ /* trace showing guardrail triggered */ ]
}
```

**Response (error):**
```json
{
  "status": "error",
  "request_id": "string, echoed",
  "error_code": "AGENT_FAILURE | LLM_TIMEOUT | INVALID_INPUT",
  "message": "human-readable string"
}
```

### 2.2 `GET /health`
Returns `{ "status": "ok", "version": "1.1" }`. Used by Railway health checks.

## 3. Agent I/O Schemas

> All agents return **strict JSON**. The OpenRouter call uses `response_format: { type: "json_object" }` where supported, plus prompt-level enforcement. The shared client validates with Pydantic; malformed JSON triggers one retry, then `AGENT_FAILURE`.

### 3.1 Guardrail Agent ⭐ NEW

**Purpose:** Detect misuse — gaming attempts, prompt injection, off-topic queries, requests for direct diagnosis or prescriptions.

**Input:**
```json
{
  "symptom_description": "string"
}
```

**Output:**
```json
{
  "should_proceed": true | false,
  "reason_category": "out_of_scope | unclear_input | prompt_injection | seeking_diagnosis | nonsensical | none",
  "confidence": "high | medium | low",
  "reasoning": "string, 1-2 sentences for logs"
}
```

**Detection categories:**
- `prompt_injection` — "ignore previous instructions", "act as a doctor", attempts to override system role
- `seeking_diagnosis` — "what disease do I have?", "diagnose me", "prescribe me X"
- `out_of_scope` — non-health queries (recipes, code help, general chat)
- `nonsensical` — clearly fake/joke inputs that aren't real health concerns
- `unclear_input` — too vague to investigate ("I feel weird")

**Routing role:** If `should_proceed=false` AND `confidence=high`, Supervisor short-circuits to `redirect` response. If `medium` confidence, log and continue with caution. If `low`, ignore.

**False positive guardrail:** Real symptoms described in vivid language ("my head feels like it's exploding") MUST pass. Prompt explicitly trains the agent on this.

### 3.2 Triage Agent

**Input:**
```json
{
  "symptom_description": "string"
}
```

**Output:**
```json
{
  "urgency_level": "emergency | urgent | routine",
  "primary_symptom_category": "string (e.g., 'cardiovascular', 'musculoskeletal', 'neurological', 'general')",
  "reasoning": "string, 1-3 sentences"
}
```

**Routing role:** Output's `urgency_level` is read by Supervisor. If `emergency`, full flow short-circuits.

### 3.3 Red Flag Detector

**Input:**
```json
{
  "symptom_description": "string"
}
```

**Output:**
```json
{
  "red_flags_detected": ["string", "..."],
  "is_emergency": true | false,
  "advisory": "string or null"
}
```

**Red flag list (v1):** chest_pain, shortness_of_breath, sudden_severe_headache, vision_loss, slurred_speech, one_sided_weakness, severe_abdominal_pain, blood_in_stool_or_vomit, suicidal_ideation, fainting.

**Routing role:** Runs in **parallel** with Triage and Guardrail. If `is_emergency=true`, Supervisor short-circuits to emergency response.

### 3.4 Symptom Deep-Dive Agent

**Input:**
```json
{
  "symptom_description": "string",
  "primary_symptom_category": "string (from Triage)",
  "previous_answers": { "q_id": "answer" }
}
```

**Output:**
```json
{
  "structured_findings": {
    "duration": "string or null",
    "severity": "mild | moderate | severe | null",
    "frequency": "string or null",
    "triggers": ["string", "..."],
    "associated_symptoms": ["string", "..."],
    "alleviating_factors": ["string", "..."]
  },
  "needs_followup": true | false,
  "followup_questions": [ /* see schema below */ ]
}
```

**Follow-up question schema (options-based, with text fallback):**

Every `followup_questions[]` entry MUST be one of these types — **never raw text-input as primary**:

```json
{
  "id": "string, e.g., 'q1_duration'",
  "question": "string, the question shown to user",
  "type": "single_choice | multi_choice | yes_no | scale",
  "choices": [
    { "value": "string", "label": "string shown to user" }
  ],
  "scale_min": 1,
  "scale_max": 10,
  "scale_min_label": "string, e.g., 'mild'",
  "scale_max_label": "string, e.g., 'severe'",
  "allow_other_text": true | false
}
```

**Rules:**
- `single_choice` and `multi_choice` use `choices[]`.
- `yes_no` uses fixed `choices: [{value:"yes",label:"Yes"},{value:"no",label:"No"}]`.
- `scale` uses `scale_min`, `scale_max`, and labels — no `choices[]`.
- `allow_other_text=true` adds an "Other (please describe)" option that opens free text. **Default: true** for `single_choice` and `multi_choice` to give users an escape hatch.
- Free-text answers are submitted as `{ "q_id_other": "user's text" }` alongside the choice value `"other"`.

**Example follow-up questions:**
```json
[
  {
    "id": "duration",
    "question": "How long have you had this symptom?",
    "type": "single_choice",
    "choices": [
      {"value": "less_than_day", "label": "Less than a day"},
      {"value": "few_days", "label": "A few days"},
      {"value": "weeks", "label": "1-3 weeks"},
      {"value": "months", "label": "More than a month"}
    ],
    "allow_other_text": true
  },
  {
    "id": "severity_scale",
    "question": "On a scale of 1-10, how severe is the pain?",
    "type": "scale",
    "scale_min": 1,
    "scale_max": 10,
    "scale_min_label": "Barely noticeable",
    "scale_max_label": "Worst imaginable"
  },
  {
    "id": "triggers",
    "question": "Does anything make it worse?",
    "type": "multi_choice",
    "choices": [
      {"value": "stress", "label": "Stress"},
      {"value": "screen_time", "label": "Screen time"},
      {"value": "physical_activity", "label": "Physical activity"},
      {"value": "specific_food", "label": "Certain foods"}
    ],
    "allow_other_text": true
  }
]
```

**Routing role:** If `needs_followup=true`, system returns `needs_followup` status. When all questions answered, Supervisor proceeds.

### 3.5 Lifestyle Agent

**Input:**
```json
{
  "symptom_description": "string",
  "deep_dive_findings": { /* structured_findings from 3.4 */ }
}
```

**Output:**
```json
{
  "lifestyle_factors": {
    "sleep_quality": "good | poor | unknown",
    "estimated_sleep_hours": "number or null",
    "screen_time_hours": "number or null",
    "stress_level": "low | moderate | high | unknown",
    "exercise_frequency": "regular | occasional | none | unknown",
    "diet_quality": "good | poor | unknown"
  },
  "lifestyle_correlations": ["string explaining how lifestyle may relate to symptom"],
  "follow_up_questions_for_doctor": ["string", "..."],
  "needs_followup": true | false,
  "followup_questions": [ /* same options-based schema as 3.4 */ ]
}
```

**Note:** Lifestyle Agent may also generate options-based follow-up questions (e.g., "How many hours do you sleep?" with choices: <5, 5-6, 7-8, 9+) following the schema in §3.4.

**Routing role:** Triggered conditionally by Supervisor (see Section 4).

## 4. Supervisor Routing Rules

> The Supervisor is **deterministic Python logic**, not an LLM call. It reads agent outputs and returns the next action. Deterministic = predictable, testable, fast. Convertible to LLM-routing in v2.

### 4.1 Routing decision tree

```
START — receives request_id + symptom_description
  │
  ├─ [Parallel] Guardrail + Triage + Red Flag Detector
  │
  ├─ IF guardrail.should_proceed == false AND guardrail.confidence == "high":
  │     → return REDIRECT response
  │     → STOP
  │
  ├─ IF red_flag.is_emergency == true:
  │     → return EMERGENCY response
  │     → STOP
  │
  ├─ IF triage.urgency_level == "emergency":
  │     → return EMERGENCY response
  │     → STOP
  │
  ├─ Symptom Deep-Dive Agent
  │
  ├─ IF deep_dive.needs_followup == true AND no follow_up_answers provided yet:
  │     → return NEEDS_FOLLOWUP to frontend
  │     → STOP (frontend re-calls with answers)
  │
  ├─ Conditional: should_run_lifestyle_agent(triage, deep_dive)?
  │     RULES:
  │     - triage.primary_symptom_category in ["general", "neurological", "musculoskeletal"] → YES
  │     - deep_dive.structured_findings.triggers contains any of ["stress", "screen_time", "screen", "work", "sleep"] → YES
  │     - deep_dive.structured_findings.associated_symptoms contains "fatigue" or "headache" → YES
  │     - else → NO
  │
  ├─ IF lifestyle decision == YES:
  │     → run Lifestyle Agent
  │     → IF lifestyle.needs_followup AND no answers yet → return NEEDS_FOLLOWUP
  │
  ├─ Assembler Agent
  │     → returns Doctor Prep Card
  │
  └─ return COMPLETE
```

### 4.2 Why this is genuinely agentic
- **9+ distinct paths** through the decision tree (redirect, emergency × 2 sources, follow-up loops, lifestyle on/off, normal completion).
- Lifestyle agent **only runs when relevant** — not always.
- Guardrail and Red Flag short-circuits demonstrate parallel reasoning.
- Different inputs produce visibly different `agent_trace` outputs.

### 4.3 Routing edge cases
- **Guardrail fails:** Treat as `should_proceed=true`. Continue. Log warning.
- **Triage fails:** Treat as `urgency_level: "routine"`, `category: "general"`. Continue.
- **Red Flag fails:** Treat as `is_emergency: false`. Continue. Log warning.
- **Deep-Dive fails:** Skip to Assembler with what we have. Card notes incomplete data.
- **Lifestyle fails:** Skip, continue to Assembler.
- **Assembler fails:** Return `error` response with raw agent outputs.

## 5. Doctor Prep Card Schema

This is the **Assembler's output** and the final user-facing artifact.

```json
{
  "summary": "string, 2-3 sentences, plain English summary",
  "symptom_timeline": {
    "primary_symptom": "string",
    "duration": "string or null",
    "severity": "string or null",
    "frequency": "string or null"
  },
  "key_findings": [
    "string, bullet-style finding"
  ],
  "lifestyle_context": "string or null, populated only if Lifestyle agent ran",
  "questions_to_ask_doctor": [
    "string, suggested question"
  ],
  "potentially_relevant_specialties": [
    "string, e.g., 'general practitioner', 'neurologist'"
  ],
  "recommended_next_step": "See a healthcare professional. [contextualized urgency message]",
  "disclaimer": "This is a preparation tool, not a diagnosis. The information here should be reviewed with a licensed medical professional. Do not delay seeking care based on this output."
}
```

**Required fields:** `summary`, `symptom_timeline`, `key_findings`, `questions_to_ask_doctor`, `recommended_next_step`, `disclaimer`.
**Optional fields:** `lifestyle_context`, `potentially_relevant_specialties`.

**Disclaimer rules:**
- The disclaimer field MUST be present and non-empty in every card.
- The default text above is the minimum. Assembler may extend but never shorten or remove.
- `recommended_next_step` MUST always recommend professional consultation. The system never tells a user "you don't need to see a doctor."

## 6. Agent Trace + Audit Logging

### 6.1 Agent trace (returned in API response)
Every response includes an `agent_trace` array showing what the system actually did. Frontend displays this so users see the agentic routing in action.

```json
{
  "agent_trace": [
    {
      "agent": "guardrail",
      "started_at": "ISO8601",
      "duration_ms": 870,
      "status": "ok",
      "decision": "should_proceed=true"
    },
    {
      "agent": "triage",
      "started_at": "ISO8601",
      "duration_ms": 1240,
      "status": "ok",
      "decision": "category=neurological, urgency=routine"
    },
    {
      "agent": "red_flag_detector",
      "duration_ms": 980,
      "status": "ok",
      "decision": "no flags"
    },
    {
      "agent": "supervisor",
      "decision": "no emergency → continue to deep_dive"
    },
    { "...": "..." }
  ]
}
```

### 6.2 Server-side audit logging

**Every event is logged to stdout in structured JSON format** (Railway captures stdout for retrieval).

**Log schema:**
```json
{
  "timestamp": "ISO8601",
  "request_id": "UUID, ties all events for one investigation",
  "event_type": "request_received | agent_started | agent_completed | agent_failed | routing_decision | response_sent | redirect_triggered",
  "agent": "string or null",
  "model": "string or null",
  "duration_ms": "number or null",
  "status": "string",
  "metadata": { /* event-specific data */ }
}
```

**What gets logged:**
- ✅ Request received (with sanitized symptom hash, NOT raw text)
- ✅ Each agent call: model, duration, success/fail, token estimates
- ✅ Each routing decision: which branch taken and why
- ✅ Final response status sent to user
- ✅ Errors with stack traces (sanitized)

**What does NOT get logged in v1:**
- ❌ Raw symptom text (avoid accidental PII leakage)
- ❌ Agent output content (logged in dev only, off in prod via env flag)
- ❌ User identifiers (anonymous in v1)

**Traceability:** Every log line has `request_id`. Filtering by `request_id` reconstructs the full event sequence for any investigation.

**v3 forward compatibility:** Log schema is designed so that when persistent storage is added, logs flow directly into a database without restructuring.

## 7. Error Handling Contract

### 7.1 OpenRouter call failures
- **Timeout:** 30s per agent. On timeout, retry once. If retry fails → that agent's status = `failed`.
- **Malformed JSON:** Validate against Pydantic. On failure, retry once with stricter prompt. If still bad → `failed`.
- **Rate limit (429):** Retry with exponential backoff (2s, 5s). Max 2 retries.
- **Other 5xx:** Retry once, then fail.

### 7.2 Per-agent failure handling
See §4.3.

### 7.3 Logging
See §6.2.

## 8. Frontend Contract

### 8.1 Screens
1. **Input screen** — single textarea + submit button. Free-text symptom entry.
2. **Loading screen** — shows `agent_trace` items as agents complete.
3. **Follow-up screen** — renders `questions[]` as **interactive option components** (buttons, sliders, multi-selects). Free-text only via "Other" fallback per §3.4.
4. **Result screen** — renders Doctor Prep Card, prominently displays disclaimer + recommended_next_step.
5. **Emergency screen** — renders advisory prominently with high-contrast warning style.
6. **Redirect screen** — renders Guardrail's redirect message, recommends professional consultation.
7. **Error screen** — renders error with retry option.

### 8.2 State machine
```
input → loading → (followup → loading)* → result | emergency | redirect | error
```

### 8.3 API integration
- Single base URL (env var: `VITE_API_BASE_URL`).
- POST `/investigate` with `Content-Type: application/json`.
- Generate `request_id` (UUID v4) on first submit; include in all subsequent calls.
- Handle all 6 response statuses.

### 8.4 Follow-up question rendering
- `single_choice` → button group (one selectable)
- `multi_choice` → checkbox group (multiple selectable)
- `yes_no` → two-button group
- `scale` → slider component with min/max labels
- `allow_other_text=true` → adds "Other" option that reveals a text input on selection

### 8.5 Visual scope (v1)
- Plain CSS, no design system.
- One color accent (e.g., teal) for primary actions.
- Mobile-friendly layout, desktop-first.
- Disclaimer always visible on result screen — never collapsed or hidden.

### 8.6 Disclaimers shown to user
- **Input screen footer:** "HealthNav is a preparation tool, not a diagnosis. Always consult a licensed medical professional."
- **Result screen prominent box:** Full disclaimer from §5.
- **Emergency screen:** Already strong by nature.
- **Redirect screen:** "Please consult a healthcare professional for personalized advice."

## 10. Future Scope — v2 and v3

> This section captures product vision beyond v1. **Architecture decisions in v1 should not paint these into a corner.**

### 10.1 v2 (estimated 15-20 hours, separate planning)
- User accounts (email/password or OAuth)
- Saved Doctor Prep Cards (user can review past cards)
- PDF export of cards
- Additional specialist agents (Stress/Mental, Medication/Allergy, Eye/Posture)
- Basic privacy policy and T&Cs page
- Email-based card sharing

### 10.2 v3 (months of work, requires legal + security review)
**Vision:** Symptom history correlation across visits — system gets "smarter" about a returning user.

**Why this is significant:** Crosses regulatory line from "wellness app" to potentially "Clinical Decision Support" (medical device classification). Requires:
- Legal counsel for T&Cs, privacy policy, liability framework
- Possibly medical device registration (FDA/CDSCO/MDR depending on jurisdiction)
- Encrypted storage of medical history
- Tamper-proof audit logs with multi-year retention
- Data deletion + export rights (GDPR, DPDP Act)
- Possibly a medical advisor on team
- Penetration testing
- Breach notification process
- Strong but legally bounded T&Cs (T&Cs cannot waive statutory rights or regulatory obligations)

**v3 cannot be shortcut by aggressive T&Cs alone.** T&Cs handle contract layer; regulatory and tort layers require actual safety work.

**v1 architectural readiness:** request_id tracing, structured logs, stateless agent design — these mean v3 adds a storage + correlation layer rather than rewriting v1.
